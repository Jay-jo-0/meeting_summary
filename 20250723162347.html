회의 핵심 요약

항목 주요 내용
회의 목표 카사코리아 AI 챗봇 개발 경험 공유를 통해 대신증권 AI 도입의 타당성 검토 및 실질적 고려사항 파악
주요 논의
  • AI 도입 인프라 (AWS Bedrock, PrivateLink, 내부망) 및 비용 (토큰 기반 과금)
  • 데이터 정제, 청크 분할, 임베딩, 벡터 DB 구축의 중요성 및 어려움
  • LLM 에이전트 설계 (LangChain, LangGraph) 및 프롬프트 엔지니어링
  • 보안 통제 (개인정보 비식별화, 감사 로그, 가드레일 한계)
  • 파운데이션 모델과 자체 개발 챗봇 성능 비교 및 할루시네이션 방지 전략
  • AI 챗봇 개발 속도 최적화 방안 (LLM 호출 최소화, 고정 답변 DB 활용)
  • 금융권 AI 도입 시 규제 및 거버넌스, 라이센스 문제
주요 결정
  • AI 활용의 타당성 검토 단계로 챗봇 프로젝트 진행 중 (내부 AI 성능 및 비용 확인 목적)
  • 개인정보 보호를 위해 LLM이 직접 데이터 학습하지 않도록 API 연동 방식 설계
  • 사용자 경험 개선을 위해 답변 속도 최적화 지속 추진 (10초 이하 목표)
향후 계획
  • AI 챗봇의 답변 품질 일관성 확보를 위한 지속적인 테스트 및 개선
  • 모바일/기획/프론트 팀과의 협업을 통한 AI 내재화 및 서비스 구체화
  • 금융 규제 당국과의 협의를 통한 AI 거버넌스 및 라이센스 확보

회의록 상세

1. 회의 서론 및 카사코리아 소개

발표자는 카사코리아 AI TF팀의 현재 개발 상황과 경험을 공유하며, 대신증권의 AI 도입에 대한 인사이트를 제공하고자 함을 밝혔다.

  • 카사코리아 개요: 부동산 조각 투자 플랫폼으로, 상업용 건물을 작은 단위로 나눠 판매하며 증권/주식처럼 공모 및 운영 배당 수익을 배분하는 구조. 5천 원 단위의 댑스(DABS) 단위로 판매. 금융위 지정 혁신 금융 기업으로 ISM-SC 인증 및 전자금융 라이센스 필요.
  • 발표자 역할: AI 도입 과정에 초점을 맞춰 인프라부터 보안까지의 경험을 공유.
  • AI 활용 예시: 문서 검색 엔진, 데이터 분석, AI 문서 요약 및 작성, AI 비서, 챗봇, AI 회의록, AI 보고서 등 (롯데 이노베이션의 아이멤버 3.0 사례 언급).

2. 챗봇 프로젝트 착수 계기 및 에이전트 AI 개념

AI 도입 시 실제 성능 검증의 필요성 때문에 챗봇 프로젝트를 시작하게 되었음을 강조했다. 특히, AI가 자연어를 얼마나 이해하는지, ChatGPT만큼 똑똑한지 확인하는 것이 중요했다.

  • 데이터 활용 전략: 외부에 공개 가능한 Q&A 데이터를 활용하여 초기 접근.
  • 에이전트 AI 정의: 이해하고 판단하며 필요한 정보를 수집 및 실행하는 AI.
    • 예시: 금융 상담 챗봇에서 고객 질문 파악(파운데이션 모델), 금융 규정 확인(리트리벌 에이전트), 요약(서머라이즈 에이전트), 권고(어드바이저 에이전트) 등 각 역할을 수행하는 AI 모듈.
    • 에이전트의 크기와 범위는 개발자의 설정에 따라 달라짐.

3. AI 도입 기술 스택 및 고려사항

3.1. 인프라 계층

카사코리아는 AWS 클라우드를 사용하며, 내부망 접속 및 보안을 최우선으로 고려하고 있다.

  • AWS 클라우드 사용: 전용선 연결 및 AWS 내부 전용 통로인 PrivateLink를 통해 AWS Bedrock 파운데이션 모델과 연결.
  • 내부망 접속: AWS 콘솔 접속 시에만 인터넷망을 사용하며, 서버 접속 및 개발 환경은 전적으로 내부망으로 구성. VPN 또는 물리적 랜선 연결을 통해서만 접근 가능.
  • AI 개발 환경: 현재 개발 중인 AI 또한 내부망으로 연결되어 외부 테스트 불가 (데모는 VPN을 통해 시연).

3.2. 모델 계층 (파운데이션 모델)

클라우드 및 온프레미스 환경에 따라 사용 가능한 파운데이션 모델이 다르며, 각 모델의 라이센스 및 구축 방식도 상이하다.

  • 모델 다양성: 각 클라우드(AWS, Oracle 등)마다 제공하는 모델이 다르며, 온프레미스 설치 모델도 다름.
  • 카사코리아 사용 모델: AWS Bedrock (SaaS 형태)을 사용하며, 사용량만큼 과금되는 시스템. 월 20~30만원 수준의 비용 발생.
  • 과금 기준: 주로 토큰 수 (수백만 건 이상), DB 종류 (PostgreSQL, OpenSearch 등)에 따라 과금.
  • 비용 효율성: 오라클 클라우드가 AWS 대비 약 3배 저렴할 수 있음 (별도 계약 여부 확인 필요).
  • 모델 활용 전략: 하나의 프로젝트에 여러 파운데이션 모델을 조합하여 사용 가능 (예: 임베딩에 Titan, 해석에 Sonnet 등). 각 에이전트 특성에 맞는 모델 선택이 중요.
  • 성능 데모: 엔트로픽의 Sonnet 모델 시연. 일반적인 질문에는 답변 가능하나, 내부 데이터가 없는 질문에는 "정보 없음" 또는 일반적인 답변을 제공.
  • 모델 선택의 중요성: 각 모델마다 성능, 특화 분야(텍스트, 이미지, 검색 등), 답변 속도, 답변 길이가 다름. 특히 PDF 내 차트/그림 인식 등은 특정 모델에 특화된 기능이 필요.

3.3. 데이터 계층

데이터 수집뿐만 아니라 정제, 분할, 임베딩, 메타데이터 관리 등 복잡한 과정이 필요하다.

  • 데이터 정제 및 청크 분할: 문서 수집 후 텍스트 정제 과정 필수. FAQ의 경우, 엑셀 한 셀을 하나의 청크로 인식하지 않고 문서 전체를 청크로 인식하여 여러 답변이 나오는 문제 발생. 이를 해결하기 위해 각 행을 개별 청크로 분할하여 임베딩.
  • 임베딩 및 벡터 DB: 인베딩 값은 사람이 알아볼 수 없는 형태로 저장되며, 질문과 답변 모두 임베딩하여 저장. 메타데이터 추가 여부, 벡터 DB 선택 등 커스텀 설정 필요.
  • 유사도 기반 신뢰도 스코어: RAG(Retrieval Augmented Generation) 활용 시 유사도 기반 신뢰도 스코어링을 통해 답변 품질 관리.
  • 데이터 관리 인력: 데이터 정제 및 관리를 위한 전담 인력 필요. 중복, 오래된 답변 등 지속적인 검증 및 수정 작업.
  • Top-K 설정: FAQ와 같이 하나의 질문에 하나의 답변만 필요한 경우 Top-K 값을 낮게 설정 (0 또는 1). 유사도 기반 검색의 경우 여러 문서를 불러오기 위해 Top-K 값을 높게 설정. 문서 유형과 접근 방식에 따라 실험을 통해 최적값 도출.

3.4. 워크플로우 계층 (개발)

LangChain, LangGraph, FastAPI, Python을 활용하여 복잡한 에이전트 워크플로우를 설계한다.

  • 프레임워크: LangChain, LangGraph, FastAPI (Python 기반).
  • 기능: 프롬프트 엔지니어링, 툴 콜링, 체인 히스토리, 스테이트 드리븐 로직 (질문 분기, 고객센터 연결, 차단 등).
  • 에이전트 설계: LLM 호출 부분을 각 에이전트 역할에 맞게 정의 (예: 질문 분석, 답변 생성 등). 현재는 간단한 형태로 구성하여 빠른 답변을 목표.
  • API 연동: 계좌/이체 API 등 내부 API와 연동하여 사용자 질문에 따라 적절한 API를 호출하고, JSON 데이터를 자연어로 변환하여 답변. LLM에게 개인정보를 직접 넘기지 않고, API 호출 결과만 정제하여 답변하도록 설계.
  • 속도 최적화: 초기 LLM을 각 스테이트마다 붙여 답변을 자연스럽게 하려 했으나, 데이터 검색(KB)에서 시간이 오래 걸리는 문제 발생 (10초 이상). 이를 해결하기 위해 고정적인 답변이 필요한 질문에 대해서는 LLM 호출 없이 FAQ DB에서 직접 답변을 가져오도록 설계. 에이전트 단위를 작게 분할하여 불필요한 데이터 검색을 줄이는 방법도 고려.
  • 테스트 및 품질 관리: 사람의 수동 테스트가 주를 이루며, 사용자 질문 패턴 분석 및 예외 처리 (인사말 인식 등)를 통해 지속적으로 개선.

3.5. 보안 통제 계층

금융권의 특성상 엄격한 보안 규제를 준수해야 하며, 개인정보 보호 및 감사 로그 기록이 필수적이다.

  • 감사 로그: 모든 입출력에 대한 감사 로그 생성 필수.
  • 개인정보 비식별화: 입력 시 개인정보(PII)가 포함될 경우 비식별화 처리 진행. AWS Guardrails와 자체 개발 필터링 로직을 병행하여 사용. Guardrails는 한글 주민등록번호, 운전면허 등 특이 케이스 인식에 한계가 있어 추가 개발 필요.
  • 응답 처리: 개인정보 포함 시 마스킹 또는 응답 차단 등 고려.
  • 망 분리 및 방화벽: 클라우드 환경에서도 망 분리 및 고가의 방화벽(예: Palo Alto) 설치, DLP(Data Loss Prevention) 등 금융 보안 솔루션 도입 필요. 이는 클라우드 자체 비용보다 훨씬 큰 금액이 발생할 수 있음.
  • AI 거버넌스: 금융 규제 당국(금융위)의 라이센스 및 승인 절차가 필요하며, AI 시스템의 법적 준수 여부를 지속적으로 확인하고 정립 중. 특히 LLM이 개인정보를 학습하지 않도록 설계하는 것이 중요.

4. 데모 및 실제 개발 경험 공유

발표자는 파운데이션 모델과 카사코리아 챗봇의 실제 동작을 시연하며, 일반적인 AI와 기업 내재화 AI의 차이점을 명확히 보여주었다.

  • 파운데이션 모델 (Sonnet) 시연: 일반적인 질문에 대한 답변은 가능하나, 내부 데이터가 없는 질문에는 "정보 없음" 또는 일반적인 지식 기반 답변을 제공.
  • 카사 챗봇 시연: 카사코리아와 관련된 질문에만 답변하며, 관련 없는 질문에는 답변을 차단. 할루시네이션(환각 현상) 방지를 위해 DB 내 질문이 없으면 일체 답변하지 않도록 설정.
  • DB 구조 시연: PostgreSQL을 사용하여 텍스트 임베딩 값이 저장되는 방식 공개. 질문과 답변이 임베딩되어 저장됨.
  • 개발팀 구성: 데이터 1명, 개발 1명, 인프라 1명, 팀장(관리) 1명으로 구성된 주니어 팀이 진행. 일반 파이썬 개발자가 일주일 정도 학습 후 개발에 참여.
  • 품질 불균일성: AI 챗봇의 답변 품질이 들쑥날쑥하며 일관성이 부족한 문제에 직면. 이는 GPT 모델 자체의 특성이기도 함.
  • AI 활용 타당성 검토: 현재 챗봇 개발은 내부 AI의 성능과 비용을 검토하는 단계로, 고객 응대 니즈보다는 기술 검증 목적이 더 큼.

5. 주요 인사이트

이번 회의를 통해 대신증권의 AI 도입에 필요한 실질적인 고려사항과 도전 과제에 대한 깊이 있는 인사이트를 얻을 수 있었다.

  • 기대와 현실의 괴리: 일반적인 AI(ChatGPT 등)에 대한 높은 기대치와 실제 기업 내재화 AI의 성능 및 구현 난이도 사이에는 상당한 격차가 존재함을 확인. 특히 금융권에서는 더욱 그러함.
  • 데이터의 중요성: AI 모델 자체보다 양질의 데이터 수집, 정제, 구조화(청크 분할, 임베딩)가 AI 챗봇 성능의 핵심임을 재확인. 데이터 전담 인력의 역할이 매우 중요함.
  • 성능 최적화의 어려움: LLM 호출 자체보다 데이터베이스에서 관련 문서를 가져오는 과정(KB 검색)이 응답 속도에 큰 영향을 미침. 이를 해결하기 위한 워크플로우 설계 및 고정 답변 활용 전략이 필수적.
  • 보안 및 규제 준수: 금융권 AI 도입 시 개인정보 비식별화, 감사 로그 기록, 망 분리 등 엄격한 보안 및 규제 준수 사항이 큰 허들로 작용. 클라우드 사용 시에도 금융 보안 솔루션 도입 비용이 상당함.
